### Critical (즉시 수정 필요) 🔴

#### 1. SessionHistory 중복 추가 가능성 ⚠️
**위치**: 
- `timerStore.ts:793` (`startBreak`)
- `TimerFullScreen.tsx:315` (`handleStopwatchComplete`)

**시나리오**:
1. 일반 타이머 시작 (집중)
2. 1분 이상 집중 후 휴식 버튼 클릭 → `startBreak()` 호출 → sessionHistory에 추가 ✅
3. 휴식 중 완료 버튼 클릭 → `handleStopwatchComplete()` 호출
4. `handleStopwatchComplete`에서 `timer.flexiblePhase === 'break_suggested' || 'break_free'`이면 마지막 세션의 breakMs만 업데이트하므로 중복은 아님 ✅
5. **하지만**: 집중 중에 완료 버튼을 누르면 `handleStopwatchComplete`에서 새 세션을 추가하는데, 이전에 휴식이 있었다면 이미 `startBreak`에서 추가되었을 수 있음

**분석 결과**:
- 코드를 보면 `handleStopwatchComplete`는 현재 세션만 처리합니다 (라인 359 주석 참조)
- 집중 중 완료: `flexiblePhase === 'focus'` → 새 세션 추가 (정상)
- 휴식 중 완료: `flexiblePhase === 'break_*'` → 마지막 세션의 breakMs만 업데이트 (정상)
- **결론**: 현재 로직상 중복 추가는 발생하지 않을 것으로 보임. ✅ **테스트 완료: 중복 없음 확인**

**테스트 완료**: [x] 집중 → 휴식 → 집중 → 완료 시나리오로 중복 확인 ✅ (2026-01-21 테스트 완료, 시나리오 1, 2, 3 모두 통과, 중복 없음 확인)

#### 3. 타이머 복원 시 시간 정확도 ✅

**시나리오**:
1. 타이머 실행 중 (`status: 'running'`, `focusStartedAt` 설정됨)

4. 이전 `focusStartedAt`과의 차이만큼 시간이 누락될 수 있음

**코드 분석**:
```typescript
if (persisted.mode === 'stopwatch' && persisted.status === 'running') {
  if (phase === 'focus' && focusStartedAt) {

    focusStartedAt = now  // 재설정
  }
}
```

**분석 결과**: 
- `focusElapsedMs`에 delta를 더하므로 누락은 없어 보임 ✅
- 하지만 `focusStartedAt`을 `now`로 재설정하면, 다음 `tick()`에서 또 delta를 더할 수 있음 ⚠️

**테스트 완료**: [x] 페이지 새로고침 후 타이머 시간 정확도 확인 ✅ (2026-01-21 테스트 완료, 모든 시나리오 성공)

  4. `resumeFocus()` (라인 816-871): 새 세션 시작 시 초기값 계산 복잡

#### 타이머 UI (`TimerFullScreen.tsx`)
- **복잡도**: 약 1130줄
- **잠재적 버그 포인트**:
  1. `handleStopwatchComplete()` (라인 281-419): 직접 store 업데이트로 상태 불일치 가능
  2. 시간 계산 로직 (라인 532-577): `sessionHistory`와 `focusElapsedMs` 동기화 복잡
  3. `handleStartBreak()` (라인 422-449): 시간 기록과 타이머 상태 변경 순서 중요

#### 통계 페이지 (`StatsPage.tsx`)
