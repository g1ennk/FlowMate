# N+1 개선 계획

## 문제
- `reorderTodos`에서 요청 아이템 수만큼 Todo를 개별 조회하여 N+1 쿼리가 발생한다.
- Todo 개수가 늘어날수록 응답 지연이 커질 가능성이 있다.

## 원인
- 서비스 로직이 `findByIdAndUserId`를 반복 호출하는 구조다.
- 재정렬 대상을 한 번에 조회/매핑하는 배치 처리 로직이 없다.

## 해결
- 재정렬 대상 ID를 모아 `IN` 조건으로 일괄 조회한다.
- 조회 결과를 `Map<id, Todo>`로 변환해 업데이트한다.
- 누락 ID는 한 번에 검증해 예외 처리한다.

## 결과(목표)
- 쿼리 수를 `N + 1`에서 `상수(2~3개)` 수준으로 줄인다.
- reorder API의 p95 응답시간 개선을 수치로 확인한다.
- 개선 전/후(쿼리 수, 응답시간)를 포트폴리오 항목으로 정리한다.
